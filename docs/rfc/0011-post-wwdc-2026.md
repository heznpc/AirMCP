## 1. Why this exists pre-conference

Multiple credible WWDC 2026 leaks signal a structural shift in Apple's AI strategy:
- **"Snow Leopard year"** — refinement-focused release cycle
- **Siri 2.0 with third-party AI extensions** — direct routing to Claude / Gemini for queries Siri can't answer ([geeky-gadgets](https://www.geeky-gadgets.com/wwdc-2026-5-new-leaks/), [TechCrunch](https://techcrunch.com/2026/03/23/apple-wwdc-june-8-12-ai-advancements-siri-developers-conference/))
- **Possible system MCP** — the developer-visible contract may finally land beyond AppIntents (RFC 0007 §1)

Any of those individually has a measurable AirMCP impact; together they could redraw the competitive map.

## 2. What this RFC will cover (to be written after the keynote)

- **System MCP API shape** — if Apple ships a first-party MCP surface, how does AirMCP coexist? Defensive options:
  - Become the "tools provider" the system MCP host consumes
  - Refocus on cross-app workflow / Skills DSL / audit (areas Apple won't ship in v1)
  - Track the new API as a transport target alongside stdio / HTTP
- **Siri AI Extensions** — Claude routing through Siri overlaps AirMCP's "talk to your Mac" pitch:
  - AirMCP's deep tool surface vs Siri's single-query handoff
  - Migration path for users who'd otherwise use Siri → Claude → MCP
- **AppIntents API additions** — any new fields (especially `requestConfirmation` API surface, MCP-specific keys) may unblock RFC 0007 Phase B
- **iOS 27 / macOS 27 SDK gates** — minimum-OS bumps and what they mean for AirMCP's compat matrix (RFC 0004)

## 5. Scenario matrix (pre-keynote prep — written 2026-05-12)

This section exists so the 48-hour post-keynote window (June 8 evening → June 10
morning KST) can be spent **executing** rather than **deciding**. Every cell
below is a hypothesis with a pre-decided response. When the keynote lands, walk
the matrix once, pick the matching quadrant, and fire the corresponding action
list. If reality doesn't match any cell, fall back to §6's escape hatch.

The two axes are the leaks with the largest AirMCP impact:

- **Axis A — System MCP API.** Does Apple ship a first-party MCP surface that
  third-party developers can register tools into (likely a superset of AppIntents
  with explicit MCP semantics, given the AppIntents-MCP convergence work the
  community has been tracking)?
- **Axis B — Siri AI Extensions.** Does Apple ship Siri-routes-to-Claude/Gemini
  for queries Siri can't answer (the leak that overlaps AirMCP's "talk to your
  Mac" pitch most directly)?

A third axis (**iWork / AppIntents API additions**) only matters within cells
where one of the first two ships, so it's threaded into each quadrant's actions
rather than expanded into a 2×2×2.

### 5.1 The four quadrants

| | **System MCP ships** | **System MCP does not ship** |
|---|---|---|
| **Siri AI Extensions ship** | **Q1 — Compete on depth.** Apple's system MCP becomes the spec everyone targets; Siri becomes the discovery layer. AirMCP's edge is the **breadth** (29 modules, ~280 tools) Apple won't ship in v1 and the **audit / Skills DSL / cross-module orchestration** Apple won't ship at all. Reposition AirMCP as "the tools provider Siri and the system MCP host consume." | **Q2 — Defend on UX.** Siri-as-router devalues the "talk to your Mac" pitch but doesn't touch the developer-facing MCP surface. AirMCP remains the only practical MCP entry point on macOS. Reposition around **developer ergonomics + cross-app workflows**; emphasize that Siri can hand off to Claude → AirMCP for anything beyond a single-shot query. |
| **Siri AI Extensions do NOT ship** | **Q3 — Become the spec target.** Best case for AirMCP. The system MCP API legitimises the protocol on Apple platforms; Apple has not given users a Siri shortcut to LLMs, so the "talk to your Mac via Claude / Cursor / Cline" UX stays AirMCP's. Action: become the **reference implementation** of the system MCP API on day one. | **Q4 — Status quo, refinement year.** "Snow Leopard" plays out: no major API shift. AirMCP continues current trajectory (RFC 0007 Phase B, RFC 0009 Phase 2, RFC 0008 Phase 2). No reposition. Use the post-WWDC window to land roadmap work that's been deferred. |

### 5.3 Threaded sub-axis — iWork / AppIntents / SDK additions

These affect every quadrant the same way and can be evaluated in parallel:

- **AppIntents `requestConfirmation` API surface added.** If Apple ships a richer
  confirmation API (beyond the 2-member `ConfirmationActionName` of iOS/macOS 26),
  RFC 0007 destructive-tool flow gets cleaner. Action: write a small follow-up
  RFC, ship behind `AIRMCP_APPINTENTS_DESTRUCTIVE_V2=true`.
- **New AppEnum primitives or typed-output channels.** Codegen helpers
  (`scripts/lib/codegen-helpers.mjs`) get extended; ineligible-tool count
  (currently 44 of ~280) shrinks. Net win regardless of quadrant.
- **iWork APIs gain scriptable surface beyond JXA.** Replace `osascript -l JavaScript`
  bridge with the new API in `src/numbers|pages|keynote/scripts.ts`. Touch RFC 0009
  to reflect the new substrate; cell-level operations get faster + better-typed.
- **macOS 27 / iOS 27 SDK gates.** If Apple announces minimum-OS bumps for
  AppIntents features, update RFC 0004's compatibility matrix and decide whether
  AirMCP's minimum supported macOS bumps too (currently 13+ for the bridge).
- **FoundationModels API additions.** Already used by `intelligence/` module
  (macOS 26+ gate). New capabilities slot into the existing module without an
  RFC; CHANGELOG entry only.

## 6. Escape hatch

If reality doesn't match any of the four quadrants (e.g. Apple ships a partial
system MCP that's read-only, or Siri AI Extensions ship as a closed beta with
only Apple-vetted LLMs, or something genuinely unexpected lands):

1. **Do not ship anything code-side in the first 48 hours.** Spend the window
   reading docs and drafting RFC 0013 instead.
2. Post a holding comms message ("We're reading the docs — full analysis in 5
   days") on README + Discussions. Do not commit to a direction publicly until
   the team has slept on it.
3. Schedule a 5-day-out follow-up: at that point, either reality looks like one
   of the four quadrants in retrospect, or this RFC gets §5.4 added to cover
   the new case.
